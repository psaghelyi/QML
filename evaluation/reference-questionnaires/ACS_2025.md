# 2025 American Community Survey: Declarative Conversion Analysis

**Source:** U.S. Census Bureau, 2025 American Community Survey (ACS), Form ACS-1(INFO)(2025), 20 pages
**QML File:** `evaluation/reference-questionnaires/ACS_2025.qml`
**Date:** 2026-03-21 (revised)

## Objective

Transform the 2025 American Community Survey paper questionnaire (20 pages, ~150 data fields, explicit GOTO-style skip instructions) into a declarative QML representation, then run the Z3-based formal validator to detect structural problems and document the design patterns exposed by the conversion.

## Methodology

1. Full reading of the 20-page PDF questionnaire (Form ACS-1(INFO)(2025))
2. Declarative QML conversion with explicit skip logic derived from the paper form's printed routing instructions ("SKIP to question X," section-gating boxes labeled A through M)
3. Formal validation using the Askalot Z3 QML validator at Level 2
4. Cross-referencing validator findings against PDF page numbers and question IDs

## Survey Architecture Overview

The ACS 2025 is a **self-administered paper questionnaire** with two major sections: a household-level Housing section and person-level demographic/socioeconomic sections repeated for up to 5 persons (with abbreviated entries for persons 6-12). The paper form uses lettered instruction boxes (A through M) and inline "SKIP to" directives for routing.

**No items are unreachable.** All 231 items have at least one valid path. The 64 ALWAYS items are fields asked of every respondent regardless of routing (household count, Person 1 basic demographics, housing building type). The 167 CONDITIONAL items are gated by block-level preconditions (person count, housing tenure, age thresholds, employment status) and item-level routing (computed variables from codeBlocks). The single CONSTRAINING postcondition validates that bedrooms cannot exceed total rooms (Q6b <= Q6a on page 8).

Note: Block-level preconditions (e.g., the housing tenure gate on rent/owner cost blocks, the employment status gate on not-working blocks, the work hours block gate on `worked_any == 1 or last_worked_recent == 1`) are propagated to all items within each block during QML loading, which is why the precondition count (167) exceeds the number of items with explicit item-level preconditions.

## Problems Exposed by Declarative Conversion

### P1: Person Roster Repetition -- Loop Structure Not Representable

**PDF evidence:** Pages 2-6 repeat identical question sets (Q1-Q6: name, relationship, sex, age/DOB, Hispanic origin, race) for Persons 1 through 5. Page 7 provides abbreviated entries (name, sex, age) for Persons 6 through 12. Page 19 states: "The balance of the questionnaire has questions for Person 2, Person 3, Person 4, and Person 5. The questions are the same as the questions for Person 1."

**Problem:** The ACS is fundamentally a **per-person repeating questionnaire** where the detailed person questions (Q7-Q44, pages 12-18) are administered identically for up to 5 persons. QML has no native loop or template mechanism. The conversion must either:
1. Unroll all 5 repetitions (creating ~750+ items for the person sections alone), or
2. Model only Person 1 as a representative template

The QML takes approach 2, modeling Person 1's full question set and the basic demographics for Persons 2-5. This means the QML represents a **partial instantiation** of the actual survey -- a single household member's complete path rather than the full multi-person form. The unrolled approach would be feasible but would produce a very large QML file with repetitive variable management (separate variables for each person's routing state).

### P2: Race Question Multi-Select Encoding -- 15 Categories Require Large Bitmask

**PDF evidence:** Q6 on pages 2-6 allows respondents to "Mark (X) one or more boxes AND print origins" across 15 race categories: White, Black or African Am., American Indian or Alaska Native, Chinese, Filipino, Asian Indian, Vietnamese, Korean, Japanese, Other Asian, Native Hawaiian, Samoan, Chamorro, Other Pacific Islander, and Some other race.

**Problem:** The ACS race question has 15 checkbox categories, requiring power-of-2 keys from 1 through 16384. This produces a theoretical maximum bitmask value of 32767 (all 15 categories selected). Several sub-categories trigger free-text follow-ups:
- White (key 1) triggers "Print origins" (German, Irish, English, etc.)
- Black (key 2) triggers "Print origins" (African American, Jamaican, etc.)
- American Indian/Alaska Native (key 4) triggers "Print tribe(s)"
- Other Asian (key 512) triggers "Print other Asian origin"
- Other Pacific Islander (key 8192) triggers "Print other PI origin"
- Some other race (key 16384) triggers "Print race or origin"

The preconditions for these follow-ups use bitwise arithmetic (e.g., `q_p1_race.outcome % 1024 >= 512` to detect "Other Asian"). This is mathematically correct but creates a **fragile coupling** between the checkbox encoding and the conditional logic. The original paper form uses simple checkbox marks with no encoding scheme -- the bitmask is an artifact of the digital representation.

### P3: Housing Section Gate Cascade -- Three-Tier Conditional Structure

**PDF evidence:** Page 8 contains instruction box **A**: "Answer questions 4-5 if this is a HOUSE OR A MOBILE HOME; otherwise, SKIP to question 6a." Page 8, Q4: "Less than 1 acre --> SKIP to question 6a." Page 10, instruction box **B**: "Answer questions 21a and b if this house, apartment, or mobile home is RENTED. Otherwise, SKIP to question 22." Page 11, instruction box **C**: "Answer questions 22-26 if you or any member of this household OWNS or IS BUYING this house, apartment, or mobile home. Otherwise, SKIP to E." Page 11, instruction box **D**: "Answer question 27 if this is a MOBILE HOME. Otherwise, SKIP to E."

**Problem:** The housing section creates a three-tier conditional cascade:
1. **Building type** (Q1) gates acreage/agriculture questions (house/mobile home only)
2. **Acreage** (Q4) gates agricultural sales (1+ acres only)
3. **Tenure** (Q20) gates rent questions (renters), owner cost questions (owners), and mobile home costs (mobile homes)

This cascade requires 5 classification variables (`is_house_or_mobile`, `has_acreage`, `is_owner`, `is_renter`, `is_mobile_home`) set via codeBlocks and consumed by block-level preconditions. The paper form handles this with simple instruction boxes (A, B, C, D), but the declarative version must explicitly model each branch. The Z3 solver confirms all branches are satisfiable and no path combinations create contradictions.

### P4: Internet Access Three-Level Skip Logic

**PDF evidence:** Page 9, Q11: "Yes, by paying a cell phone company or Internet service provider" (no skip). "Yes, without paying a cell phone company or Internet service provider --> SKIP to question 13." "No access to the Internet at this house, apartment, or mobile home --> SKIP to question 13."

**Problem:** Q11 (Internet access) has three response options, but only one (paid access, option 1) leads to the detailed Internet service type questions (Q12a-e). The other two options both skip to Q13. In the paper form, this is a simple "SKIP to question 13" instruction. In the declarative model, the five Internet service sub-questions (Q12a-e: cellular, broadband, satellite, dial-up, other) each need a precondition on `internet_paid == 1`.

Additionally, Q12e ("some other service") has its own conditional follow-up for specification text, creating a nested conditional: Internet paid --> other service selected --> specify. This two-level nesting is straightforward in QML but represents implicit complexity that the paper form handles with a single skip instruction.

### P5: Employment Status Decision Tree -- Seven-Node Routing Graph

**PDF evidence:** Pages 16-17 contain a complex employment routing structure:
- Q30a: "Did this person work for pay?" Yes --> SKIP to Q31. No --> Q30b.
- Q30b: "Did this person do ANY work?" Yes --> SKIP to Q31. No --> Q36a.
- Q36a: "On layoff?" Yes --> SKIP to Q36c. No --> Q36b.
- Q36b: "Temporarily absent?" Yes --> SKIP to Q39. No --> Q37.
- Q36c: "Recalled?" Yes --> SKIP to Q38. No --> Q37.
- Q37: "Looking for work?" Yes --> Q38. No --> SKIP to Q39.
- Instruction **L**: "Answer questions 36-39 if this person did NOT work last week. Otherwise, SKIP to question 40a."

**Problem:** This is the most complex routing structure in the ACS, with 7 branching nodes and 5 distinct paths through the employment status questions. The paper form uses a cascade of "SKIP to" instructions that create a decision tree. Converting this to declarative preconditions requires 6 classification variables (`worked_last_week`, `worked_any`, `did_not_work`, `on_layoff`, `temp_absent`, `looked_for_work`) set incrementally as the respondent proceeds through the tree.

The key challenge is that the "not working" path (Q36-Q39) is gated on BOTH `did_not_work == 1` AND `q_30b_any_work.outcome == 2` (No). The paper form's sequential SKIP logic handles this implicitly (you only reach Q36 if you answered No to both Q30a and Q30b), but the declarative model must explicitly encode both conditions as a block-level precondition. The `q_30b_any_work.outcome` reference creates a dependency on an item that may itself be skipped (if `did_not_work == 0`), but the Z3 solver confirms this is satisfiable because the block precondition requires `did_not_work == 1`, which guarantees Q30b was shown.

### P6: Utility Cost Questions -- Repeated Pattern with "OR" Alternatives

**PDF evidence:** Page 10, Q17a-d each follow the same pattern: enter a dollar amount, OR select "Included in rent or condominium fee" / "No charge or not used." For example, Q17a (electricity): dollar amount box, OR "Included in rent or condominium fee," OR "No charge or electricity not used."

**Problem:** The paper form uses a visual "OR" separator between the dollar amount write-in and the alternative checkboxes. This is not a skip pattern -- it is a **mutually exclusive input mode** where the respondent either enters a number or selects a categorical alternative. QML has no native "enter a number OR select a category" construct. The conversion models this as a Radio question (choose mode) followed by a conditional Editbox (enter amount if "dollar amount" was selected). This doubles the number of items: 4 utility questions become 8 items (4 Radio + 4 Editbox).

The same pattern appears in Q23 (property taxes), Q24 (insurance), Q25b (mortgage payment), Q26b (second mortgage payment), and Q44 (total income), adding 10 more items. The paper form's visual "OR" layout is compact and intuitive, but the digital representation requires explicit branching for each instance.

### P7: Age-Gated Sections -- Five Distinct Age Thresholds

**PDF evidence:**
- Page 14, instruction box **H**: "Answer questions 19a-c if this person is 5 years old or over. Otherwise, SKIP to the questions for Person 2 on page 19."
- Page 14, instruction box **I**: "Answer question 20 if this person is 15 years old or over. Otherwise, SKIP to the questions for Person 2 on page 19."
- Page 15, instruction box **J**: "Answer question 25 if this person is female and 15-50 years old. Otherwise, SKIP to question 26a."
- The marital status questions (Q21-Q24) are implicitly gated on age 15+ (same instruction box I).
- Q20 (independent living difficulty) is gated on age 15+.

**Problem:** The ACS uses 5 distinct age thresholds that gate different question sections:
1. Age 0+ (no gate) -- hearing, vision disability
2. Age 5+ -- cognitive, ambulatory, self-care disability
3. Age 15+ -- independent living, marital status, employment
4. Female AND age 15-50 -- fertility
5. No explicit age gate -- grandparent questions, military service

The declarative model computes these gates in Person 1's age codeBlock using three variables (`is_age_5plus`, `is_age_15plus`, `is_female_15_50`). The conversion requires careful attention to the boundary values: the PDF says "5 years old or over" (>= 5), "15 years old or over" (>= 15), and "15-50 years old" (>= 15 AND <= 50). An off-by-one error in any threshold would incorrectly include or exclude respondents.

Note that the employment section (Q30-Q44) has no explicit age gate in the paper form, even though employment questions are typically only relevant for working-age persons. The ACS asks these questions of all persons regardless of age, which means a 5-year-old would theoretically receive employment questions. In practice, the answer to Q30a ("Did this person work for pay?") would be "No" for young children, routing them through the not-working path.

### P8: "Specify Year" Sub-Questions -- Conditional Write-In Fields

**PDF evidence:** Page 8, Q2: "2020 or later -- Specify year" with a write-in box. Page 12, Q10b: "Grade 1 through 12 -- Specify grade 1-12" with a write-in box. Page 12, Q11: "Grade 1 through 11 -- Specify grade 1-11" with a write-in box. Page 8, Q8: "Yes, U.S. citizen by naturalization -- Print year of naturalization" with a write-in box.

**Problem:** Several Radio/Dropdown questions include a write-in sub-field for one specific option. The paper form integrates these visually (the write-in box appears next to the option), but QML must model them as separate conditional items. This creates a pattern where a Dropdown item is followed by a conditional Editbox:

```
Q2 (year built) = "2020 or later" --> Q2_specify (Editbox, min 2020, max 2025)
Q10b (grade level) = "Grade 1-12" --> Q10b_specify (Editbox, min 1, max 12)
Q11 (highest degree) = "Grade 1-11" --> Q11_specify (Editbox, min 1, max 11)
Q8 (citizenship) = "Naturalized" --> Q8_year (Editbox, min 1900, max 2025)
```

Each "specify" sub-question is a separate item with its own precondition, increasing the total item count. The paper form's inline write-in fields are more compact.

## Impact Assessment

| Category | Paper Form (PDF) | Declarative (QML) |
|----------|------------------|-------------------|
| **Person repetition** | Same questions printed 5 times (pages 2-6, 12-18 x5) | Single-person template -- reveals non-representable loop |
| **Race multi-select** | Simple checkboxes with 15 categories | Power-of-2 bitmask (1-16384) with bitwise preconditions |
| **Housing conditional cascade** | Instruction boxes A-D with "SKIP to" | 5 classification variables + block-level preconditions |
| **Internet skip logic** | "SKIP to question 13" on 2 of 3 options | `internet_paid` variable gates 5 sub-items |
| **Employment routing** | 7-node decision tree with cascading SKIPs | 6 classification variables with nested preconditions |
| **Utility cost "OR" pattern** | Visual "enter amount OR select alternative" | Radio (choose mode) + conditional Editbox, doubling items |
| **Age thresholds** | Instruction boxes H, I, J with age gates | 3 computed variables from single age codeBlock |
| **Specify sub-fields** | Inline write-in next to specific option | Separate conditional Editbox items |

## Conclusion

The Z3 QML validator found **no structural defects** (no cycles, no unreachable items, no infeasible postconditions) in the ACS 2025 conversion. All 231 items are reachable (64 ALWAYS, 167 CONDITIONAL), and the single postcondition (bedrooms <= total rooms) is confirmed as CONSTRAINING (neither always true nor always false).

The declarative conversion exposed **8 categories of design observations**:

1. **Person roster repetition** -- the ACS's fundamental per-person loop structure (pages 2-6 for basic demographics, pages 12-18 repeated for up to 5 persons as noted on page 19) cannot be represented natively in QML, requiring either a single-person template or massive unrolling
2. **Race question bitmask complexity** -- 15 race categories on pages 2-6 require power-of-2 keys up to 16384, with bitwise arithmetic for 6 conditional free-text follow-ups
3. **Housing conditional cascade** -- instruction boxes A through D on pages 8 and 10-11 create a three-tier branching structure requiring 5 classification variables
4. **Internet access three-level skip** -- Q11 on page 9 has three options where only one leads to detailed sub-questions, requiring an `internet_paid` gate variable
5. **Employment decision tree** -- pages 16-17 contain a 7-node decision tree (Q30a/b, Q36a/b/c, Q37, Q38) requiring 6 classification variables for declarative representation
6. **Utility cost "OR" pattern** -- Q17a-d on page 10 and Q23-Q26 on page 11 use a visual "amount OR alternative" layout that doubles the item count when split into Radio + Editbox pairs
7. **Five age thresholds** -- instruction boxes H, I, J on pages 14-15 use ages 5+, 15+, and female 15-50 to gate disability, marital, and fertility sections
8. **Specify sub-fields** -- Q2, Q8, Q10b, Q11 on pages 8 and 12 include inline write-in fields that become separate conditional items in QML

The ACS 2025 is a well-structured self-administered questionnaire where most routing logic is explicit (printed "SKIP to" instructions). Unlike CATI booklets where routing is entirely implicit, the ACS's paper-form skip patterns translate cleanly to QML preconditions. The primary complexity arises from the per-person repetition pattern and the large number of classification variables needed to model the housing tenure and employment status decision trees.
